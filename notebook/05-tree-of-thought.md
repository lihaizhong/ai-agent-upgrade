# 思维树 (Tree of Thoughts, ToT)

## 核心概念

思维树（Tree of Thoughts, ToT）是由 Yao 等人（2023）提出的一种高级提示工程技术，用于解决需要探索或预判战略的复杂任务。它基于思维链（Chain-of-Thought）提示，引导语言模型探索把思维作为中间步骤来解决通用问题。

### 思维树的定义

ToT 维护着一棵思维树，其中：
- **思维（Thought）**：由连贯的语言序列表示，是解决问题的中间步骤
- **树结构**：思维以树的形式组织，每个节点代表一个思维步骤
- **探索能力**：模型能够自己对严谨推理过程的中间思维进行评估

### 工作原理

```
初始问题
    ↓
生成多个思维（候选项）
    ↓
评估每个思维的质量
    ↓
选择最优思维（或保留多个）
    ↓
基于选定的思维生成新思维
    ↓
重复评估和选择
    ↓
最终解决方案
```

## ToT 框架原理

### 核心机制

ToT 框架将以下能力结合：

1. **思维生成**：语言模型能够生成多个不同的思维候选项
2. **思维评估**：模型能够评估每个思维的质量和可行性
3. **搜索算法**：结合 BFS、DFS 或 Beam Search 等搜索算法
4. **前瞻验证**：在探索过程中可以向前验证和回溯

### 框架图解

```
问题
  ├─ 思维 A
  │   ├─ 思维 A1
  │   │   ├─ 思维 A1.1 → 评估
  │   │   └─ 思维 A1.2 → 评估
  │   └─ 思维 A2
  │       └─ ...
  ├─ 思维 B
  │   ├─ 思维 B1
  │   └─ 思维 B2
  └─ 思维 C
      ├─ 思维 C1
      └─ 思维 C2
```

## 应用示例：算 24 游戏

### 任务描述

"算 24 游戏"是一个数学推理任务，给定 4 个数字，需要通过加减乘除运算得到 24。

### ToT 配置

- **思维步骤数量**：3 个思维步骤
- **每步候选项数量**：保留最优的 5 个候选项
- **搜索算法**：广度优先搜索（BFS）

### 思维生成

每个步骤需要生成一个中间方程，例如：

**步骤 1：**
- 候选项 1：`4 + 6 = 10`
- 候选项 2：`9 - 4 = 5`
- 候选项 3：`6 * 4 = 24`
- ...

**步骤 2：**（基于步骤 1 的结果）
- 如果选择 `6 * 4 = 24`，已经得到 24
- 如果选择 `4 + 6 = 10`，需要继续：`10 + 5 = 15`
- ...

**步骤 3：**（最终验证）

### 思维评估

每步思维都要求 LM 给出评估：

**评估等级：**
- **sure**（一定能）：经过少量向前尝试就可以验证正确
- **maybe**（可能）：无法确定，需要继续探索
- **impossible**（不可能）：基于"太大/太小"的常识可以排除

**评估过程：**
1. 生成 3 个评估结果（抽样）
2. 综合评估结果决定是否保留该思维
3. 消除那些"不可能"的局部解
4. 保留"maybe"的局部解继续探索

### 示例流程

假设数字是：`4, 6, 9, 5`

**步骤 1：生成初始思维**
```
思维 1: 6 * 4 = 24 → 评估: sure
思维 2: 9 + 5 = 14 → 评估: maybe
思维 3: 9 - 5 = 4 → 评估: maybe
思维 4: 6 + 4 = 10 → 评估: maybe
思维 5: 9 / 3 = 3 → 评估: impossible（3 不在输入中）
```

**步骤 2：基于 sure 的思维（已找到解）**
```
如果选择 6 * 4 = 24：
  评估: sure → 找到解！
```

**步骤 2：基于 maybe 的思维继续探索**
```
如果选择 9 + 5 = 14：
  可用数字: 14, 4, 6
  新思维: 14 + 6 + 4 = 24 → 评估: sure
```

## 搜索算法

### 1. 广度优先搜索（BFS）

**原理：**
- 逐层探索所有可能的思维
- 保留每层最优的 N 个候选项
- 直到找到解或达到深度限制

**优点：**
- 保证找到最优解（如果存在）
- 探索全面

**缺点：**
- 计算成本高
- 可能需要大量资源

**适用场景：**
- 解空间较小
- 需要最优解

### 2. 深度优先搜索（DFS）

**原理：**
- 沿着一个思维路径深入探索
- 直到找到解或达到深度限制
- 然后回溯探索其他路径

**优点：**
- 内存效率高
- 快速找到可行解

**缺点：**
- 可能不是最优解
- 可能陷入死胡同

**适用场景：**
- 解空间较大
- 需要快速找到解

### 3. 集束搜索（Beam Search）

**原理：**
- 结合 BFS 和 DFS
- 每层只保留最优的 K 个候选项
- 平衡探索和利用

**优点：**
- 计算效率高
- 解质量较好

**缺点：**
- 可能错过最优解
- 需要调整 K 值

**适用场景：**
- 解空间中等
- 需要平衡质量和效率

## 性能对比

根据 Yao 等人（2023）的报告，ToT 在"算 24 游戏"任务上的表现：

| 方法 | 成功率 | 特点 |
|------|--------|------|
| **标准 CoT** | ~50% | 单一路径推理 |
| **自我一致性** | ~70% | 多路径投票 |
| **思维树（ToT）** | ~90%+ | 系统性探索 |

**结论：** ToT 大大超过了其他提示方法。

## ToT 与其他方法的对比

### Yao et al. (2023) vs Long (2023)

#### 相似点

两种方法的核心思路类似：
- 都以多轮对话搜索树的形式增强 LLM
- 都能够解决复杂问题
- 都需要回溯和探索

#### 不同点

| 特性 | Yao et al. (2023) | Long (2023) |
|------|------------------|-------------|
| **搜索策略** | DFS/BFS/Beam | 强化学习训练的 ToT 控制器 |
| **策略类型** | 通用搜索策略 | 针对具体问题优化 |
| **学习能力** | 静态策略 | 可以从新数据学习 |
| **进化能力** | 有限 | 可以通过自对弈不断进化 |
| **适用性** | 通用任务 | 特定任务优化 |

#### 强化学习控制器的优势

- **自适应**：可以根据任务调整搜索策略
- **学习能力**：可以从新数据集中学习
- **自对弈**：类似 AlphaGo，可以通过自我对弈改进
- **持续进化**：即使使用冻结的 LLM，系统仍可进化

## ToT 提示词方法

### 简化版 ToT 提示

Hulbert（2023）提出了一种将 ToT 框架概念概括成简短提示词的方法：

```
假设三位不同的专家来回答这个问题。
所有专家都写下他们思考这个问题的第一个步骤，然后与大家分享。
然后，所有专家都写下他们思考的下一个步骤并分享。
以此类推，直到所有专家写完他们思考的所有步骤。
只要大家发现有专家的步骤出错了，就让这位专家离开。
请问...
```

### 提示词解析

1. **三位专家**：模拟多个思维路径
2. **逐步分享**：逐步展示推理过程
3. **错误排除**：评估并排除错误路径
4. **最终答案**：基于剩余路径得出结论

### 优势

- **简单易用**：单次提示即可
- **无需复杂实现**：不需要编程实现搜索算法
- **灵活性高**：适用于各种问题

### 局限性

- **控制精度低**：无法精确控制搜索策略
- **资源效率低**：可能生成不必要的思维
- **评估粗糙**：缺乏精细的评估机制

ToT 特别适用于：

- ✅ **复杂推理任务** - 需要多步推理的问题
- ✅ **探索性任务** - 需要探索多种可能性的任务
- ✅ **战略规划** - 需要预判和规划的情境

- ✅ **创意生成** - 需要探索多种创意方向的场景

## 局限性

### 1. 计算成本高

- 需要生成大量思维
- 每个思维都需要评估
- 搜索过程消耗资源

### 2. 实现复杂

- 需要实现搜索算法
- 需要设计评估机制
- 需要管理思维树

### 3. 任务依赖

- 需要针对不同任务调整
- 评估标准需要定制
- 参数需要调优

### 4. 上下文限制

- 思维树可能很大
- 可能超出模型上下文窗口
- 需要压缩或分批处理

## 实现示例

### 基础 ToT 实现

```python
class TreeOfThoughts:
    def __init__(self, max_branches=5, max_depth=3):
        self.max_branches = max_branches
        self.max_depth = max_depth
        self.thoughts = []
    
    def generate_thoughts(self, problem, current_thought=None):
        """生成思维候选项"""
        prompt = f"""
        问题：{problem}
        {'当前思维：' + current_thought if current_thought else ''}
        
        请生成 {self.max_branches} 个不同的下一步思维：
        """
        thoughts = call_llm(prompt)
        return thoughts.split('\n')
    
    def evaluate_thought(self, thought):
        """评估思维"""
        prompt = f"""
        评估以下思维，给出评估等级（sure/maybe/impossible）：
        {thought}
        
        评估：
        """
        evaluation = call_llm(prompt)
        return evaluation.strip()
    
    def search(self, problem):
        """使用 BFS 搜索解"""
        queue = [(problem, [])]  # (当前状态, 思维路径)
        
        while queue:
            current_state, path = queue.pop(0)
            
            # 检查是否找到解
            if self.is_solution(current_state):
                return path
            
            # 限制深度
            if len(path) >= self.max_depth:
                continue
            
            # 生成思维
            thoughts = self.generate_thoughts(current_state, path[-1] if path else None)
            
            # 评估并筛选
            for thought in thoughts:
                evaluation = self.evaluate_thought(thought)
                
                if evaluation in ['sure', 'maybe']:
                    new_state = self.apply_thought(current_state, thought)
                    new_path = path + [thought]
                    queue.append((new_state, new_path))
        
        return None  # 未找到解
```

## 评估指标

### 1. 任务性能

- **成功率**：找到解的概率
- **解质量**：解的最优性
- **求解时间**：找到解所需时间

### 2. 资源效率

- **思维生成数**：生成的思维总数
- **评估次数**：思维评估的总次数
- **计算成本**：API 调用成本

### 3. 搜索效率

- **探索深度**：搜索树的最大深度
- **分支因子**：平均分支数
- **剪枝效果**：无效思维的剪除率

## 常见问题

### Q1：如何确定思维步骤的数量？

根据任务复杂度：
- 简单任务：2-3 步
- 中等任务：3-5 步
- 复杂任务：5-10 步

### Q2：如何选择候选项数量？

平衡探索和效率：
- 资源充足：5-10 个
- 资源有限：3-5 个
- 快速原型：2-3 个

### Q3：ToT 适用于所有任务吗？

不适用于：
- 简单任务（过度设计）
- 创意任务（需要多样性）
- 实时任务（延迟要求高）

### Q4：如何处理思维评估的不确定性？

可以：
- 多次抽样评估
- 使用加权评分
- 考虑评估的置信度
