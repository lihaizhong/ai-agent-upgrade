# 少样本提示 (Few-Shot Prompting)

## 核心概念

少样本提示是一种在提示中提供演示（示例）的技术，通过上下文学习引导模型实现更好的性能。演示作为后续示例的条件，帮助模型理解任务并生成期望的响应。

## 工作原理

### 为什么需要少样本提示

虽然大型语言模型展示了惊人的零样本能力，但在更复杂的任务上，零样本设置仍然表现不佳。少样本提示通过以下方式提升性能：

1. **上下文学习** - 提供示例让模型学习任务模式
2. **条件引导** - 演示作为后续示例的条件
3. **模式识别** - 帮助模型识别任务的结构和格式

### 模型规模效应

根据 Touvron et al. (2023) 和 Kaplan et al. (2020) 的研究：
- 当模型规模足够大时，少样本提示特性开始出现
- 大型模型更能够从少量示例中学习

## 示例

### 示例 1：学习新词使用（1-shot）

**提示：**

```
"whatpu"是坦桑尼亚的一种小型毛茸茸的动物。一个使用whatpu这个词的句子的例子是：
我们在非洲旅行时看到了这些非常可爱的whatpus。
"farduddle"是指快速跳上跳下。一个使用farduddle这个词的句子的例子是：
```

**输出：**

```
当我们赢得比赛时，我们都开始庆祝跳跃。
```

**分析：** 模型通过提供一个示例（1-shot）学会了如何执行任务。

### 示例 2：增加演示数量

对于更困难的任务，可以增加演示数量：
- **1-shot**：1 个示例
- **3-shot**：3 个示例
- **5-shot**：5 个示例
- **10-shot**：10 个示例

### 示例 3：随机标签（格式的重要性）

**提示：**

```
这太棒了！// Negative
这太糟糕了！// Positive
哇，那部电影太棒了！// Positive
多么可怕的节目！//
```

**输出：**

```
Negative
```

**关键发现：** 即使标签已随机化，模型仍然得到了正确答案。这表明：

- 格式对性能起着关键作用
- 使用随机标签比没有标签好得多
- 从真实标签分布中选择随机标签也有帮助

### 示例 4：格式不一致的稳健性

**提示：**

```
Positive This is awesome!
This is bad! Negative
Wow that movie was rad!
Positive
What a horrible show! --
```

**输出：**

```
Negative
```

**观察：** 即使格式不一致，模型仍然预测了正确的标签。新的 GPT 模型对随机格式变得更加稳健。

## 最佳实践（Min et al., 2022）

根据研究，以下是在进行少样本学习时的关键提示：

### 1. 标签空间和输入分布很重要
- 演示指定的输入文本的分布很重要
- 无论标签是否对单个输入正确，分布信息都有帮助

### 2. 格式的作用
- 使用的格式对性能起着关键作用
- 保持一致的格式有助于模型理解

### 3. 标签分布
- 从真实标签分布（而不是均匀分布）中选择随机标签
- 这比随机分配标签效果更好

## 适用场景

少样本提示适用于：

- ✅ 需要特定模式或格式的任务
- ✅ 复杂任务，零样本效果不佳
- ✅ 需要精确控制输出格式
- ✅ 需要更高的输出一致性
- ✅ 教授模型新的概念或用法

## 局限性

### 少样本提示的限制

标准的少样本提示对许多任务都有效，但仍然不是完美的技术，特别是在处理更复杂的推理任务时。

### 示例：推理任务的失败

**任务：** 判断奇数之和是否为偶数

**零样本提示：**

```
这组数字中的奇数加起来是一个偶数：15、32、5、13、82、7、1。
A：
```

**输出：**

```
是的，这组数字中的奇数加起来是107，是一个偶数。
```

**问题：** 107 不是偶数，答案错误。

**少样本提示：**

```
这组数字中的奇数加起来是一个偶数：4、8、9、15、12、2、1。
A：答案是False。
这组数字中的奇数加起来是一个偶数：17、10、19、4、8、12、24。
A：答案是True。
这组数字中的奇数加起来是一个偶数：16、11、14、4、8、13、24。
A：答案是True。
这组数字中的奇数加起来是一个偶数：17、9、10、12、13、4、2。
A：答案是False。
这组数字中的奇数加起来是一个偶数：15、32、5、13、82、7、1。
A：
```

**输出：**

```
答案是True。
```

**结果：** 仍然错误。少样本提示不足以获得这种类型的推理问题的可靠响应。

**原因：**
- 任务涉及多个推理步骤
- 需要将问题分解成步骤并向模型演示
- 简单的示例不足以教会模型复杂的推理过程

## 何时使用更高级的技术

当零样本提示和少样本提示不足时，意味着模型学到的东西不足以在任务上表现良好。此时应该考虑：

### 1. 思维链提示 (Chain-of-Thought Prompting)
- 将问题分解成步骤
- 向模型演示推理过程
- 适用于复杂的算术、常识和符号推理任务

### 2. 微调模型
- 在特定任务上微调模型
- 适用于需要大量领域知识的任务

### 3. 其他高级提示技术
- 角色扮演提示
- 自我一致性提示
- 等等

## 与零样本提示的对比

| 特性 | 零样本提示 | 少样本提示 |
|------|-----------|-----------|
| 示例数量 | 0 个 | 1 个或多个 |
| 适用任务 | 简单、通用任务 | 复杂、特定任务 |
| 输出质量 | 中等 | 较高 |
| 一致性 | 较低 | 较高 |
| 提示长度 | 短 | 较长 |
| 推理能力 | 有限 | 有所改善 |

## 设计少样本提示的技巧

### 1. 选择代表性示例
- 选择能代表任务多样性的示例
- 覆盖不同的输入分布
- 包含边界情况

### 2. 保持格式一致
- 使用统一的格式
- 清晰标记输入和输出
- 保持分隔符一致

### 3. 合理的示例数量
- 从少量开始（1-3 个）
- 根据任务复杂度增加
- 避免过多示例导致混淆

### 4. 示例顺序
- 从简单到复杂
- 从典型到边缘情况
- 考虑示例之间的关联性

## 常见错误

### 1. 示例质量差
- 错误的示例会误导模型
- 确保示例的正确性

### 2. 格式不一致
- 混乱的格式降低性能
- 保持格式的一致性

### 3. 示例数量不当
- 太少：学习不足
- 太多：上下文占用过多

### 4. 示例分布不均
- 覆盖面不够
- 导致模型对某些情况不熟悉

## 实践建议

1. **从零样本开始**：先尝试零样本，评估效果
2. **逐步添加示例**：从 1-shot 开始，逐步增加
3. **优化示例选择**：选择最具代表性的示例
4. **测试不同格式**：尝试不同的格式和分隔符
5. **评估性能**：对比不同配置的效果
6. **考虑上下文长度**：示例数量受模型上下文窗口限制

## 相关技术

- **零样本提示（Zero-Shot Prompting）**：不提供示例
- **思维链提示（Chain-of-Thought Prompting）**：引导逐步推理
- **自我一致性（Self-Consistency）**：多次采样取共识
- **思维树（Tree-of-Thought）**：探索多种推理路径
