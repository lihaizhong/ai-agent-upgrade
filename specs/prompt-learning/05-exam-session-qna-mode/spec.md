# Spec: Prompt Learning Exam Session Q&A Mode

## 前置依赖

1. `01-learning-platform-rearchitecture` 已将 `prompt-learning` 重构为平台型产品，考试中心成为独立模块
2. `03-selector-first-interaction` 已明确选择类交互优先走执行器原生选择器
3. `04-interaction-mode-declaration` 已为考试入口、考试结构和考试蓝图补齐 `interaction` 声明

## 问题陈述

当前考试中心已经具备：

- 固定考试类型入口
- 固定题位蓝图
- 单题结构校验
- 单题评分能力
- 终局报告生成
- 考试历史回写

但它仍缺少“考试会话层”。

这导致当前考试更接近“生成整张试卷后一次性展示”，而不是控制台环境更适合的“逐题问答式考试”。在控制台中，一次性展示整卷会带来以下问题：

- 用户需要在终端里来回滚动，注意力容易被打断
- LLM 必须自行记忆当前做到了哪一题，状态不稳定
- 无法严格表达“上一题提交后锁定、不可回看”的考试约束
- 中断后难以恢复，或恢复时缺少明确的产品规则
- 考试场景和练习场景的反馈节奏容易混淆

## 目标

为考试中心新增“逐题 Q&A 会话模式”，使 `diagnostic` 与 `final` 两种考试都遵循统一的控制台考试流程：

1. 一次只展示一道题
2. 用户提交答案后，该题立即锁定
3. 上一题不可回看
4. 考试进行中不公布即时反馈、对错或得分
5. 考试结束后统一公布报告、解析、薄弱点和建议
6. 考试中断后，若存在未完成会话，用户可自主选择“恢复考试”或“放弃考试”
7. 同一时刻同一用户只允许一个进行中的考试会话

## 非目标

本规格不包括：

- 自适应考试或动态难度曲线
- 跳题、标记题目、回看上一题
- 在考试过程中展示即时反馈
- 引入新的考试类型
- 改变现有考试题数、题型、分值和蓝图结构
- 将完整题干全文长期持久化到 workspace

## 核心决策

### 1. `diagnostic` 和 `final` 同时切到逐题模式

这次改动针对的是“控制台考试交互方式”，不是“考试目的”。

- `diagnostic`：诊断考试，快速识别知识缺口
- `final`：综合考试，检查系统性掌握程度

两者都属于考试场景，应共享同一套逐题会话模型，而不是一个逐题、一个整卷，避免产品心智分裂。

### 2. 上一题不可回看

题目一旦提交即锁定：

- 不允许重新展示上一题供修改答案
- 不允许在会话中回滚当前题号
- 不允许在恢复会话时回到更早的已提交题目

### 3. 反馈只在终局统一公布

考试过程中：

- 不展示“正确 / 错误”
- 不展示即时得分
- 不展示逐题解析

考试结束后统一给出：

- 总分与分项得分
- 每题简洁解析
- 薄弱课程
- 薄弱主题
- 下一步学习建议

### 4. 中断后由用户选择恢复或放弃

若用户已有 `in_progress` 会话，再次进入考试中心时必须优先处理该会话：

- 选 `恢复考试`：回到当前未完成题
- 选 `放弃考试`：将该会话标记为 `abandoned`

放弃考试不应写入正式考试成绩。

## 用户流程

目标流程如下：

1. 用户进入考试中心
2. 选择考试类型（`diagnostic` / `final`）
3. 脚本检查是否存在未完成考试会话
4. 若存在未完成会话，优先询问“恢复”或“放弃”
5. 若无未完成会话或用户选择放弃旧会话，则创建新会话
6. 返回当前题位信息，由 LLM 按题位生成当前题内容
7. 用户提交当前题答案
8. 脚本记录答案并锁定该题，不返回即时反馈
9. 若仍有下一题，则返回下一题题位信息
10. 若已完成最后一题，则统一评分、生成报告、回写考试历史

## CLI 范围

本规格允许为 `exam` 模块新增命令。

建议新增：

- `exam --start --type <diagnostic|final>`
- `exam --current-question --session <id>`
- `exam --submit-answer --session <id>`
- `exam --resume`
- `exam --abandon --session <id>`
- `exam --finish --session <id>`

也可根据实现需要合并为更少命令，但最终必须满足以下能力边界：

1. 能创建考试会话
2. 能返回当前题上下文
3. 能提交并锁定当前题答案
4. 能恢复未完成会话
5. 能放弃未完成会话
6. 能完成考试并生成终局结果

## 会话状态模型

新增考试会话状态，建议放在 `prompt-learning-workspace/<username>/exam/` 下。

建议字段：

```json
{
  "session_id": "exam-2026-04-09T20-30-00",
  "exam_type": "diagnostic",
  "status": "in_progress",
  "current_question_num": 3,
  "total_questions": 11,
  "started_at": "2026-04-09T20:30:00+08:00",
  "updated_at": "2026-04-09T20:35:00+08:00",
  "completed_at": null,
  "answers": [
    {"num": 1, "type": "mc", "answer": "B"},
    {"num": 2, "type": "mc", "answer": "D"}
  ],
  "scores": [],
  "question_summaries": [
    {"num": 1, "type": "mc", "goal": "基础概念识别"},
    {"num": 2, "type": "mc", "goal": "简单场景判断"}
  ]
}
```

状态枚举：

- `in_progress`
- `completed`
- `abandoned`

## 持久化约束

本规格必须遵守现有 workspace 的克制持久化原则。

允许持久化：

- 会话状态
- 当前题号
- 用户答案
- 分数摘要
- 会话状态变更时间
- 题目摘要或题位引用

不允许默认持久化：

- 整张试卷全文
- 每题完整题干全文
- 冗余对话日志
- LLM 中间推理

如果实现上需要短期缓存当前题完整题面，也必须确保它只服务于进行中的会话，不作为长期学习档案的一部分。

## 交互模式要求

新增考试会话相关输出时，必须继续遵守 `interaction` 协议：

- 恢复/放弃选择：`selector`
- 考试入口类型选择：`selector`
- 当前题展示：`inform`
- 终局报告：`inform`

题目本身不是选择器元数据；只有用户需要做分支选择的点，才应使用 `selector`。

## 成功标准

本规格完成当且仅当：

1. `diagnostic` 和 `final` 都以逐题模式运行
2. 同一用户同一时刻最多只有一个 `in_progress` 考试会话
3. 用户提交上一题后无法回看或修改
4. 考试过程中不返回即时反馈、得分或解析
5. 考试结束后能统一生成报告并回写正式考试历史
6. 中断后再次进入考试中心时，用户能明确选择“恢复考试”或“放弃考试”
7. 放弃考试不会写入正式考试成绩

## 风险

1. **状态复杂度上升**：考试从“无状态出卷”变成“有状态会话”后，CLI 和持久化都要扩展
2. **题面缓存边界模糊**：逐题恢复可能需要当前题内容，但现有原则不鼓励持久化完整题干
3. **LLM 与脚本职责混淆**：若脚本不托管会话，LLM 会重新承担状态管理，导致设计失效
4. **放弃逻辑污染历史**：若 abandoned 会话被错误计入正式成绩，会破坏学习档案可信度

## 开放问题

1. 恢复会话时，当前题是否应重新展示完整题面，还是由脚本返回缓存后的当前题内容？本规格倾向于允许恢复当前题完整题面，但仅限进行中会话使用
2. 大题评分是否完全依赖脚本输入的 rubric scores，还是允许 LLM 先生成 rubric scores 再交给脚本汇总？本规格允许保留现状，不强制在本次 change 内解决
