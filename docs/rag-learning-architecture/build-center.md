# Build Center 设计

## 目标

将原有“实战搭建”能力重构为平台中的 `实战中心`，使其承担“带着用户做出可运行系统”的职责，而不是停留在讨论方案。

实战中心的核心目标是：

- 帮用户跑通一个最小 RAG
- 让关键技术选择按顺序落地
- 每轮只推进一个关键决策
- 把学习中心的框架转成真实实现

## 核心定位

实战中心是平台中的“交付模块”。

它负责：

- 场景与约束澄清
- 项目分阶段推进
- 当前决策推荐
- 最小实现任务拆解
- 结果摘要记录

它不负责：

- 深度变量对比实验
- 企业方案全面评审

这些分别由 RAG Lab 和架构评审承担。

## V1 项目定位

V1 只要求完整做通一个项目：

- `本地最小 RAG`

它应覆盖以下最小步骤：

1. 场景与成功标准
2. 文档导入与分块
3. Embedding 选择
4. 向量库选择
5. 检索实现
6. 是否引入 rerank
7. 回答生成与引用
8. 基础评估

## 每轮输出协议

每一轮都按下面格式组织：

1. `当前决策`
2. `推荐方案`
3. `取舍理由`
4. `最小实现任务`
5. `进入下一轮的条件`

这样能保证用户知道当前只需要做什么，而不是被整套架构压垮。

## 项目推进原则

### 1. 一次只解决一个关键问题

不要在同一轮同时讨论 embedding、向量库和 rerank。

### 2. 优先做能证明价值的最小路径

先跑通检索和回答，再考虑优化层。

### 3. 能不加复杂度就不加

例如在数据量较小、质量尚可时，不默认引入 rerank。

### 4. 关键分歧用实验解决

当某个组件选择争议较大时，应引导用户进入 RAG Lab，而不是继续空谈。

## 与 RAG Lab 的衔接

以下情况优先建议进入实验：

- 用户纠结两种 embedding
- 不确定是否需要 rerank
- 调 chunk size / top-k 没有把握
- 召回质量与延迟存在明显冲突

实验完成后，再回到当前步骤继续推进。

## 与架构评审的衔接

以下情况应建议切到架构评审：

- 用户明确是企业项目
- 已出现多租户、安全、权限、可用性、治理等问题
- 目标不再是“先跑通”，而是“正式方案”

## CLI 建议

实战中心对应 `build` 模块，建议支持：

- `build --entry-points`
- `build --resume`
- `build --scenario <name>`
- `build --step-panel --project <id> --step <name>`
- `build --record-step`
- `build --summary`

## 状态更新建议

每完成一个关键步骤，应至少更新：

- `current_module = build`
- `current_project`
- `last_action`
- `recommended_next_action`
- `build-progress.json`

## 数据职责划分

### 脚本负责

- 项目入口
- 步骤结构
- 当前步骤面板
- 进度写入
- 项目摘要聚合

### LLM 负责

- 约束澄清
- 当前决策推荐
- 取舍说明
- 代码级指导
- 下一步解释

## V1 成功标准

实战中心设计成功后，应满足：

- 用户能跑通一个本地最小 RAG
- 用户知道当前做到哪一步
- 用户能把关键选择说清楚
- 遇到分歧时知道何时进入实验或评审
