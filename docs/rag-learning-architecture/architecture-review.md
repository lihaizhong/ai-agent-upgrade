# Architecture Review 设计

## 目标

将零散的企业级问答和方案讨论重构为平台中的 `架构评审` 模块，使其承担“从业务约束生成结构化 RAG 方案”的职责。

架构评审的核心目标是：

- 把需求说清楚
- 把推荐方案讲清楚
- 把权衡和风险显式化
- 把后续评估与优化路线写出来

## 核心定位

架构评审是平台中的“方案输出模块”。

它负责：

- 业务场景梳理
- 关键约束澄清
- 推荐架构输出
- 组件选型与理由
- 风险与评估建议

它不负责：

- 详细编码实现
- 逐轮项目推进
- 大量变量实验

## 适用场景

适合以下请求：

- 企业知识库搜索方案
- 客服机器人架构设计
- 多租户 RAG 方案讨论
- 私有部署 / 安全合规场景
- 成本、性能、稳定性权衡

## 核心流程

建议固定为以下流程：

1. 明确业务场景
2. 收集约束
3. 识别关键决策点
4. 输出推荐架构
5. 说明组件选型
6. 显式列出风险
7. 给出评估与上线建议
8. 用户确认是否保存摘要

## V1 评审模板

V1 模板建议固定以下字段：

- `业务场景`
- `目标用户`
- `数据来源`
- `语言与规模`
- `部署方式`
- `成功标准`
- `推荐架构`
- `组件选型`
- `取舍理由`
- `主要风险`
- `评估方案`
- `上线后优化路线`

## 约束澄清原则

### 一次只补最关键的 1 到 2 个约束

不要把评审做成冗长问卷。

### 先问改变最大决策的约束

例如：

- 是否必须私有部署
- 数据规模是千级还是百万级
- 是否必须保留来源引用

### 约束不足时不装作确定

如果关键条件缺失，应明确说明推荐是临时方案。

## 输出原则

### 推荐必须默认给出一个主方案

不要只列 5 个候选项。

### 权衡必须显式

至少应回答：

- 为什么这样选
- 为了什么收益
- 付出了什么代价

### 风险必须具体

例如：

- 缺评估集
- 文档质量差
- 权限模型未定义
- 成本不可控

而不是笼统写“有一定风险”。

## 与实战中心的关系

当用户只是想“先跑通系统”时，不应直接拉入评审。

当用户已经出现以下诉求时，应优先进入评审：

- 企业落地
- 多团队使用
- 合规与权限
- 成本与 SLA
- 上线前方案对齐

## 与 RAG Lab 的关系

评审可引用实验结果，但不要求每次评审都先做实验。

如果某个关键选型争议较大，评审可建议：

- 先做 `embedding` 实验
- 先做 `rerank` 实验
- 先做 `chunking` 实验

## CLI 建议

架构评审对应 `review` 模块，建议支持：

- `review --entry-points`
- `review --template --scenario <name>`
- `review --record-summary`
- `review --history`

## 数据职责划分

### 脚本负责

- 评审模板
- 结构化字段
- 历史记录写入
- 历史摘要展示

### LLM 负责

- 约束澄清
- 推荐方案论证
- 风险说明
- 优化路线表达

## V1 范围

架构评审 V1 先做：

- 单份通用评审模板
- 结构化方案输出
- 历史摘要记录

暂不做：

- 自动成本计算器
- 自动容量规划器
- 多方案并排评分系统
