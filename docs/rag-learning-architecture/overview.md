# RAG Learning 平台化重构总览

## 背景

当前 `rag-learning` skill 已具备课程讲解、实战引导和专题问答三类能力，但整体仍更接近“带路由的知识型 skill”，而不是一个连续的 RAG 学习与搭建平台。

当前主要问题：

- 用户心智仍停留在“学习 / 实战 / 专题问答”三条路径，而不是统一产品
- `SKILL.md` 同时承担产品定位、交互规则、流程控制和时效性提醒，职责过重
- 课程、最小 RAG 搭建、技术选型、方案评审之间缺少统一状态流转
- 缺少持续实验与对比记录，用户容易停留在“知道组件名词”，而不是“形成工程判断”
- 企业级用户缺少稳定的方案评审入口，无法把单点问答沉淀为结构化架构输出

## 重构目标

将 `rag-learning` 从“RAG 学习与问答 skill”重构为“RAG 系统设计训练平台”，形成统一入口、统一状态、统一持久化和统一实验记录。

重构后应满足：

- 用户进入后优先看到平台首页，而不是底层路由模式
- 平台能够同时承接三类结果：
  - 跑通本地最小 RAG
  - 完成 embedding / rerank / 向量库等关键选型
  - 输出企业级 RAG 方案草案
- `SKILL.md` 只定义 agent contract，不再承载流程细节和状态 schema
- 脚本负责导航、状态、实验结构、模板输出和持久化
- LLM 负责讲解、追问、点评、方案论证和实验结论表达

## 产品心智模型

重构后的 `rag-learning` 是一个训练平台，内部包含五个模块：

- 学习中心
- 实战中心
- RAG Lab
- 架构评审
- 学习档案

平台首页负责统一导航，默认展示以下核心动作：

- 继续学习
- 搭建最小 RAG
- 进入 RAG Lab
- 发起架构评审

学习档案作为次级入口，用于展示进度、实验历史、选型偏好和近期建议。

## 模块定义

### 学习中心

负责建立统一的 RAG 选型框架：

- 场景与成功标准
- 数据形态与分块
- Embedding 选择
- 向量库选择
- 检索策略与混合检索
- Rerank 选择
- 生成层设计
- 评估与调优

### 实战中心

负责“带着做”一个最小可运行 RAG：

- 明确场景与约束
- 按决策顺序完成实现
- 每轮只推进一个关键选择
- 形成能跑通的最小系统

### RAG Lab

负责变量实验与对比分析：

- Embedding 对比
- Rerank 是否值得引入
- Chunking / Top-K / 混合检索实验
- 记录结果并沉淀结论

### 架构评审

负责企业级方案输出：

- 约束澄清
- 推荐架构
- 组件选型
- 权衡与风险
- 评估方案
- 上线后优化路线

### 学习档案

负责聚合长期状态：

- 当前模块与当前项目
- 已完成课程与实战（含准确的 active / completed 计数）
- 实验摘要
- 选型偏好（从实验 `recommended_choice` 与评审 `recommended_stack` 自动 rollup）
- 架构评审历史

## 设计原则

### 1. 平台优先于路由

用户体验应围绕训练平台展开，而不是围绕“学习 / 实战 / 问答”三选一路由展开。

### 2. 决策优先于知识罗列

平台的重点不是列举所有模型、数据库和框架，而是帮助用户建立稳定的判断顺序和取舍方法。

### 3. Script-First

脚本负责：

- 首页导航
- 状态流转
- 课程目录
- 实战步骤结构
- 实验蓝图
- 架构评审模板
- 持久化读写

LLM 负责：

- 教学讲解
- 约束澄清
- 动态练习与实验内容
- 方案论证
- 反馈与下一步建议

### 4. 实验驱动，而不是资料堆叠

用户应通过比较实验和最小实现获得经验，而不是只阅读课程或 FAQ。

### 5. 持久化克制

只保留长期有价值的信息，不保存冗余问答日志、中间推理或临时草稿。

## 持久化策略

每个用户使用独立 workspace：

`rag-learning-workspace/<username>/`

workspace 仅保存真正需要长期保留的信息，例如：

- 当前模块与当前项目摘要
- 课程进度
- 最小 RAG 搭建进度
- 实验历史与结论
- 选型偏好
- 架构评审摘要

## 职责划分

### `SKILL.md`

只保留：

- 产品定位
- 角色定义
- 行为准则
- 模块路由规则
- 脚本调用边界
- 持久化边界
- 禁止事项

### `scripts/`

承接：

- 平台首页
- workspace 初始化
- 状态管理
- 学习中心结构
- 实战中心结构
- RAG Lab 蓝图
- 架构评审模板
- 学习档案聚合

### `courses/`

只负责课程内容源，不承担状态和流程。

### `reference/`

只保留补充材料和轻量映射说明，不再作为主流程驱动源。

## 建议的正式模块边界

后续脚本层建议稳定为：

- `workspace`
- `home`
- `learning`
- `build`
- `lab`
- `review`
- `profile`

## 成功标准

重构完成后，系统应满足：

- 用户始终知道当前处于哪个模块
- 用户能围绕固定决策顺序学习和搭建 RAG
- 用户能完成至少一个本地最小 RAG
- 用户能做出可解释的组件选型
- 用户能输出结构化企业级方案
- 平台能稳定记录实验和方案历史
