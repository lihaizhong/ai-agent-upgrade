# RAG Learning 平台化迁移计划

## 目标

将当前 `rag-learning` 从轻路由 skill 逐步迁移为平台型 skill，同时控制改动风险，避免一次性重写全部课程、参考资料和交互流程。

## 迁移原则

### 1. 先稳定产品心智，再扩脚本

优先确定模块边界、状态模型和 skill contract，再决定脚本与持久化实现。

### 2. 先做一个闭环，再扩场景

先做通：

- 首页
- 学习中心主线课程
- 本地最小 RAG
- 3 个基础实验
- 通用评审模板

不要一开始同时覆盖所有场景。

### 3. 保留课程资产，重做组织方式

现有 `courses/` 和部分 `reference/` 仍可复用，但要重新纳入平台模块，而不是继续直接暴露为静态目录。

## 建议阶段

## 阶段 1：平台文档与 contract

产出：

- `docs/rag-learning-architecture/`
- 新版 `SKILL.md` 草案

目标：

- 确认产品心智
- 确认模块边界
- 确认持久化边界

## 阶段 2：脚本层骨架

产出：

- `workspace`
- `home`
- `learning`
- `build`
- `lab`
- `review`
- `profile`

目标：

- 打通首页与状态流转
- 提供结构化输出骨架

## 阶段 3：主线课程与最小实战

产出：

- 学习中心主线课程包
- `本地最小 RAG` 实战闭环

目标：

- 证明用户能“先学框架，再落地实现”

## 阶段 4：RAG Lab 与评审沉淀

产出：

- 3 个基础实验蓝图
- 通用架构评审模板
- 历史记录摘要

目标：

- 建立“实验驱动 + 方案输出”的平台差异化

## 阶段 5：旧结构清理

产出：

- 精简 `reference/`
- 清理过时描述
- 将旧“学习 / 实战 / 专题问答”心智从 `SKILL.md` 中移除

目标：

- 避免新旧产品心智并存

## 风险点

- 如果先写大量课程而不做状态与模块骨架，平台会再次退化成资料库
- 如果先做过重脚本而没有验证产品节奏，会增加维护负担
- 如果不处理时效性问题，Embedding / Rerank / 向量库相关内容会快速过时

## MVP 完成标志

当以下条件同时满足时，可认为平台化 MVP 完成：

- 存在平台首页
- 用户可进入学习中心并完成一节主线课程
- 用户可在实战中心跑通一个最小 RAG
- 用户可在 RAG Lab 中完成至少一个实验并保存结论
- 用户可在架构评审中输出一份结构化方案摘要
- 用户可在学习档案中看到自己的当前状态与下一步建议
