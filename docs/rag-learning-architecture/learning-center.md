# Learning Center 设计

## 目标

将现有课程能力重构为平台中的 `学习中心`，让用户围绕“决策框架”学习 RAG，而不是按组件清单被动吸收知识。

学习中心的核心目标是：

- 帮用户建立稳定的 RAG 选型顺序
- 让课程讲解形成统一结构
- 让课程自然衔接实战与实验
- 让用户完成从概念到判断的迁移

## 核心定位

学习中心是平台中的“方法论模块”。

它负责：

- 课程选择
- 学习路径推荐
- 课程讲授
- 课后迁移问题
- 向实战和实验分流

它不负责：

- 完整项目推进
- 大量变量实验
- 企业方案输出

这些分别由实战中心、RAG Lab 和架构评审承担。

## 学习中心主流程

建议固定为以下闭环：

1. 进入学习中心
2. 查看推荐学习包或课程目录
3. 选择课程
4. 获取课程元数据与文件路径
5. LLM 读取课程内容并完成讲解
6. 用 1 个迁移问题确认理解
7. 引导进入：
   - 最小实战
   - 对应实验
   - 下一节课

## 课程组织原则

学习中心不再以“按组件平铺目录”为主，而是以“决策顺序”组织主线。

建议主线顺序：

1. 场景与成功标准
2. 数据形态与分块
3. Embedding 选择
4. 向量库选择
5. 检索策略与混合检索
6. Rerank 决策
7. 生成层与引用策略
8. 评估与调优
9. 企业级架构与治理

## 课程讲解结构

课程讲解必须由 LLM 组织输出，不直接粘贴课程原文。

建议统一采用以下结构：

1. 一句话理解
2. 这个决策解决什么问题
3. 判断时看哪些约束
4. 常见选项与边界
5. 一个最小场景例子
6. 一个企业场景例子
7. 常见误区

这个结构的重点不是“解释定义”，而是“告诉用户怎么判断”。

## 课后迁移问题

每节课讲完后，应有一个轻量迁移问题，验证用户是否能把当前知识迁移到真实场景。

示例方向：

- 如果你的数据从 Markdown 变成 PDF，当前决策会先变哪里
- 如果成本优先而不是效果优先，你会改哪一层
- 什么时候先不加 rerank

V1 保持 1 个问题即可，不做重型测试。

## 与实战中心的衔接

学习中心默认不是以“继续下一课”为唯一出口。

讲完以下课程后，优先推荐：

- `数据形态与分块` -> 进入实战中的 chunking 步骤
- `Embedding 选择` -> 进入实战中的 embedding 步骤
- `向量库选择` -> 进入实战中的 vector DB 步骤
- `Rerank 决策` -> 进入 RAG Lab 做对比实验
- `评估与调优` -> 进入实验或架构评审

## 与状态模型的关系

学习中心至少应更新这些状态：

- `current_module = learning`
- `current_course_id`
- `last_action`
- `recommended_next_action`

常见状态转换：

- 进入课程：`lesson_started`
- 讲解完成：`lesson_completed`
- 课程完成：`course_completed`

## CLI 建议

学习中心对应 `learning` 模块，建议支持：

- `learning --catalog`
- `learning --path --level <novice|intermediate|advanced|enterprise>`
- `learning --recommend-course`
- `learning --lesson-meta --course <N>`
- `learning --lesson-panel --course <N>`
- `learning --complete --course <N>`

## 数据职责划分

### 脚本负责

- 课程目录
- 路径元数据
- 课程文件路径
- 课后面板
- 状态更新

### LLM 负责

- 课程讲解
- 迁移问题
- 启发式答疑
- 场景映射

## V1 范围

学习中心 V1 先做：

- 主线课程目录
- 推荐学习包
- 课程讲授闭环
- 迁移问题
- 到实战或实验的分流

暂不做：

- 更细的小节级进度追踪
- 多学习路径并行地图
- 独立考试体系
