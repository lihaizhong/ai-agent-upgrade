# RAG Learning Platform

## Problem Statement
How might we 帮初中级工程师和企业级进阶用户，先建立稳定的 RAG 选型框架，再通过分层实战与评估闭环，完成从最小 RAG 搭建到企业级方案输出的能力跃迁？

## Recommended Direction
把 `rag-learning` 改造成一个双轨制教学平台，而不是继续做“学习/实战/专题问答”的轻路由 skill。平台的核心不是讲更多 RAG 知识，而是训练三种能力：`会搭`、`会选`、`会评`。这要求它同时具备结构化学习、实验驱动实践、企业方案评审三种能力。

推荐方向是：`选型驱动型 + 实验室型 + 评审型` 的组合平台。用户先进入统一平台首页，再按当前目标进入学习中心、实战中心、RAG Lab、架构评审、学习档案。学习中心负责建立组件决策框架，实战中心负责完成最小可运行 RAG，RAG Lab 负责做 embedding/rerank/chunking/vector DB/hybrid search 对比实验，架构评审负责把能力迁移到真实业务约束下的企业方案设计。这样既能服务“想搭一个能跑的系统”的人，也能服务“要做企业级选型和架构输出”的人。

## Key Assumptions to Validate
- [ ] 用户真正需要的是“选型方法 + 实验闭环”，不是更多 RAG 名词解释。验证方式：先用 5-10 个典型请求跑新路由，看用户是否更常选择 `实战中心` 和 `RAG Lab`，而不是只停在问答。
- [ ] `prompt-learning` 式平台结构能提升 RAG 学习连续性。验证方式：先做最小首页、模块路由和进度记录，观察是否能支撑“继续上次实验/继续上次项目”。
- [ ] 实验驱动比纯课程目录更能建立工程经验。验证方式：先设计 3 个基础实验包，比较“读课程”与“做实验”两种路径的完成率与后续提问质量。
- [ ] 企业级用户愿意用 skill 做方案评审和技术选型草案。验证方式：提供 `架构评审` 入口，测试是否能稳定产出包含约束、权衡、风险、评估指标的方案草稿。

## MVP Scope
MVP 不做“全量 RAG 平台”，只做最小闭环：

1. 平台首页
统一入口到 `学习中心`、`实战中心`、`RAG Lab`、`架构评审`、`学习档案`。

2. 学习中心
先只上线一条主线课程：`RAG 选型框架`。
内容不是按组件罗列，而是按决策顺序组织：
`场景与成功标准 → 数据与分块 → Embedding → 向量库 → 检索策略 → Rerank → Generation → Evaluation`

3. 实战中心
先只做一个项目：`本地最小 RAG`
要求用户完成：
`文档导入 → 分块 → Embedding → 本地向量库 → 检索 → 回答生成 → 基础评估`

4. RAG Lab
先做 3 个实验：
`Embedding 对比`
`Rerank 要不要加`
`Chunk size / top-k 对结果影响`
每个实验都输出固定记录：目标、变量、结果、结论、下一步建议。

5. 架构评审
先做一个固定输出模板：
`业务场景`
`约束`
`推荐架构`
`组件选型`
`权衡`
`风险`
`评估方案`
`上线后优化路线`

6. 学习档案
只持久化有长期价值的信息：
当前模块、已完成实验、已完成项目、已确认的选型偏好、历史架构评审摘要。

## Not Doing (and Why)
- 不做“所有 RAG 话题的百科全书”。
原因：这会把平台重新做回资料库。
- 不做“默认一次性输出完整教程/完整代码仓架”。
原因：用户会看完但学不会，且交互节奏失控。
- 不先做多场景项目矩阵。
原因：MVP 先证明一个最小 RAG 项目和一套实验闭环即可。
- 不把选型建议写死成静态推荐榜单。
原因：Embedding、Rerank、向量库都高度时效性，必须强调决策框架和官方资料校验。
- 不一上来复制 `prompt-learning` 的全部脚本复杂度。
原因：RAG 平台先验证模块价值，再决定哪些能力值得脚本化。

## Open Questions
- `rag-learning` 是否也要像 `prompt-learning` 一样引入脚本层来承载首页、状态、实验记录和推荐逻辑？
- MVP 的“本地最小 RAG”默认技术栈要不要固定，例如 `Python + LangChain/LlamaIndex + Qdrant/Chroma + OpenAI/Open-source embeddings`？
- 企业级方案输出是否需要单独沉淀到 `docs/rag-learning-architecture/` 之类的设计文档目录？
