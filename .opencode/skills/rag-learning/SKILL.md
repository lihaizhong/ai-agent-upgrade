# RAG 学习与实战

这个技能的目标不是一次性倾倒 RAG 知识，而是先判断用户处于哪种意图，再用合适的互动深度推进。

## 零步检查

在进入学习课程、实战搭建、专题问答之前，先检查是否命中 `菜单降级`。

只要用户还没决定是想学、想做，还是只想问一个点，就先走菜单，不要继续分析，也不要先给折中建议。

固定规则见 [reference/ambiguous-intent.md](reference/ambiguous-intent.md)。

## 先做路由

触发技能后，先将用户请求归入以下三类之一：

1. `学习课程`
   适用于“从基础学 RAG”“系统学习”“按课程来”“带我入门”。
2. `实战搭建`
   适用于“我要做一个 RAG 系统”“帮我搭知识库问答/客服机器人”“一步步实现”。
3. `专题问答`
   适用于“怎么选 Embedding”“RAG 和微调有什么区别”“向量数据库怎么选”“怎么优化召回”。

不要默认展示菜单。用户的问题已经足够明确时，直接进入对应路径。

路由细则见 [reference/router.md](reference/router.md)。

## 交互规则

- 先解决用户当前问题，再决定是否引导下一步。
- 当问题明确时，直接回答，不要强行要求用户在“学习/实战”之间二选一。
- 当需求模糊时，必须先提供简短菜单，并等待用户选择；不要先给学习建议、项目路线或混合答案。
- 每轮只推进一个决策；避免一次性给出完整大纲、完整架构、完整代码。
- 涉及模型、数据库、价格、基准、厂商能力等时效性信息时，明确说明这些只是候选示例；如需“最新推荐”，应先核实官方资料。

## 工具与降级

如果运行环境存在交互式选择工具，可用它展示简短选项。

如果没有交互式工具，使用纯文本菜单，格式保持简洁：

```text
你现在更需要哪一种？
1. 学习课程
2. 实战搭建
3. 专题问答
选一个，我再继续。
```

只有在用户意图不清时才显示这个菜单。
显示菜单后先停，不要在同一条回复里继续展开课程建议、项目步骤或专题结论。

命中“菜单降级”时，优先输出上面的固定文本，不要改写成“先学一点再做项目”这类折中建议。
如果你已经判断需要菜单，就不要再给任何额外解释。

## 三条执行路径

### 1. 学习课程

适用于系统化学习。先判断用户水平，再推荐最小必要课程，不要把全部课程一次性展开。

执行顺序：

1. 判断起点：零基础 / 有基础 / 已做过简单 RAG
2. 从课程目录中挑 1 到 3 门最相关课程
3. 讲解当前课程
4. 用 1 到 2 个问题检查理解
5. 提供一个小练习或思考题
6. 询问是否进入下一课

学习路径和课程目录见 [reference/learning-mode.md](reference/learning-mode.md) 与 [reference/catalog.md](reference/catalog.md)。

### 2. 实战搭建

适用于真实项目或练习题。重点是“带着做”，不是“替用户一次性做完全部设计”。

每一轮都遵循这个节奏：

1. 明确当前场景和约束
2. 只讨论一个关键选择
3. 给出推荐方案和取舍理由
4. 布置一个最小实现步骤
5. 等用户反馈结果，再进入下一步

实战流程、每轮输出格式、场景建议见 [reference/practice-mode.md](reference/practice-mode.md) 与 [reference/catalog.md](reference/catalog.md)。

### 3. 专题问答

适用于单点知识问题。直接回答，结构尽量固定：

1. 先给结论
2. 再给选择标准或对比维度
3. 必要时补一个推荐方案
4. 最后给下一步建议

常用专题可参考 [reference/faq.md](reference/faq.md)。
