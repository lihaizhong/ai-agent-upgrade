# RAG Skill 路由规则

先判断用户到底是在学、在做，还是只想问一个点。判断错了，后面内容再完整也会显得啰嗦或跑题。

在做下面这些判断前，先看 [ambiguous-intent.md](ambiguous-intent.md)。如果命中模糊意图，就先走菜单，不要继续往下路由。

## 路由到学习课程

满足以下信号时，进入 `学习课程`：

- 用户说“从基础开始学”“系统学习”“给我安排课程”“带我入门”
- 用户没有具体项目，只是想建立完整知识体系
- 用户愿意按章节循序推进

**响应方式**：
- 不要先讲全部知识点
- 先判断基础，再推荐 1 到 3 节最合适的课

## 路由到实战搭建

满足以下信号时，进入 `实战搭建`：

- 用户说“我要做一个 RAG 系统”“帮我搭建”“一步步实现”
- 用户给出明确场景，如文档问答、客服机器人、企业知识库
- 用户关心组件选择、代码实现、测试和调优

**响应方式**：
- 先收集约束：数据类型、语言、预算、部署方式、目标效果
- 每轮只解决一个关键设计决策

## 路由到专题问答

满足以下信号时，进入 `专题问答`：

- 用户只问一个点，如“向量数据库怎么选”“RAG 和微调区别”
- 用户想快速得到推荐，而不是进入完整课程或项目流程
- 用户请求比较 Embedding、Rerank、Chunking、评估指标等

**响应方式**：
- 直接回答
- 用结论、对比维度、推荐方案三段式组织

## 需要菜单的情况

只有下面两种情况才弹菜单：

1. 用户只说“我想了解 RAG”，但没说明想学、想做还是想问
2. 用户同时表达多个方向，无法确定当前最优切入点

菜单最多三项，而且必须先让用户做选择：

```text
你现在更需要哪一种？
1. 学习课程
2. 实战搭建
3. 专题问答
选一个，我再继续。
```

菜单回复只做这件事：

- 不要先给“先学一点再做项目”这类混合建议
- 不要先展开学习路线
- 不要先展开项目方案
- 等用户选完，再进入对应路径

命中菜单时，建议直接输出这段，不要自由发挥：

```text
你现在更需要哪一种？
1. 学习课程
2. 实战搭建
3. 专题问答
选一个，我再继续。
```

## 不要这样做

- 不要把“如何选择 Embedding 模型”这类问题先改写成模式选择题
- 不要对明确的项目需求先输出完整课程目录
- 不要对初学者默认推荐企业级架构
