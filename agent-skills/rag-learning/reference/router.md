# RAG 平台模块路由说明

这个文件不再承担旧“学习 / 实战 / 专题问答”产品壳的职责，而是补充说明平台模块如何承接不同用户意图。

实际产品入口以平台首页和脚本模块为准。

## 路由原则

先判断用户当前最需要推进哪一个平台模块，而不是先让用户理解内部实现方式。

优先模块：

1. `学习中心`
2. `实战中心`
3. `RAG Lab`
4. `架构评审`
5. `学习档案`

## 路由到学习中心

满足以下信号时，进入 `学习中心`：

- 用户说“从基础开始学”“系统学习”“建立选型框架”“给我安排课程”
- 用户没有明确项目，只是想建立方法论
- 用户愿意按课程或学习包循序推进

响应方式：

- 不要先讲全部知识点
- 先判断当前水平，再推荐最小学习包
- 优先围绕决策顺序讲解，而不是组件名词清单

## 路由到实战中心

满足以下信号时，进入 `实战中心`：

- 用户说“我要做一个 RAG 系统”“帮我搭建”“一步步实现”
- 用户给出明确场景，如文档问答、客服机器人、企业搜索
- 用户关心组件选择、实现步骤和验证方法

响应方式：

- 先收集当前决策最关键的 1 到 2 个约束
- 每轮只解决一个关键设计决策
- 当有明显选型分歧时，建议进入 `RAG Lab`

## 路由到 RAG Lab

满足以下信号时，进入 `RAG Lab`：

- 用户想比较 embedding / rerank / chunking / top-k
- 用户说“我想做个实验看看差异”
- 用户已经在实战中推进，但某个组件选择存在不确定性

响应方式：

- 一次只比较一个变量
- 固定实验条件和指标
- 必须记录实验结论并回流到实战或评审

## 路由到架构评审

满足以下信号时，进入 `架构评审`：

- 用户说“给我一份企业方案”“帮我评审架构”
- 用户关心私有部署、多租户、权限、安全、SLA、成本
- 用户已经超出“先跑通系统”的范围

响应方式：

- 先补最关键约束
- 给一个主方案，而不是只列候选项
- 显式说明取舍、风险和评估方式

## 路由到学习档案

满足以下信号时，进入 `学习档案`：

- 用户说“看进度”“看实验记录”“看历史方案”“看我学到哪了”

响应方式：

- 展示当前状态、近期实验、近期评审和下一步推荐

## 需求模糊时

如果用户只表达“想了解 RAG”，但没有说明想学、想做、想实验还是想出方案，应优先回到平台首页，而不是先给旧式三选一路由。
