# RAG 实战模式详细说明

## 概述

实战模式面向“我要做一个 RAG 系统”的请求。核心原则是逐步推进：每一轮只处理一个关键决策，避免一次把完整方案全部抛给用户。

场景列表统一见 [catalog.md](catalog.md)。

## 进入实战前先确认

正式给方案前，先尽量补齐这些约束：

- 场景：文档问答、客服机器人、企业搜索、垂直领域等
- 数据：PDF、Markdown、网页、数据库，或混合数据
- 语言：中文、英文、多语言
- 规模：几百条、几万条、还是百万级
- 预算：优先效果、优先成本，还是可本地部署
- 部署方式：本地验证、云服务、企业内网
- 成功标准：更准、更快、更便宜，还是更可解释

如果用户没给全，不要一次问很多。优先问当前决策最关键的 1 到 2 个约束。

## 每轮输出协议

每一轮都按下面格式组织：

1. `当前决策`
   只说本轮要决定什么，例如 Embedding、向量库、分块策略。
2. `推荐方案`
   给出一个默认推荐，而不是只列一堆选项。
3. `取舍理由`
   用语言、规模、成本、部署方式解释为什么这样选。
4. `最小实现任务`
   给用户一个当前就能完成的小步骤。
5. `进入下一轮的条件`
   告诉用户做完这一步后，下一步该继续什么。

## 推荐推进顺序

### 第 1 轮：选场景与成功标准

先确认用户是在做什么系统，以及最优先目标是什么。

### 第 2 轮：选 Embedding 与分块策略

优先根据语言、数据类型和准确率要求来定，不要脱离场景单独讨论。

候选示例：
- 通用云服务：`text-embedding-3-small`
- 更重中文或多语言：`bge-m3`
- 本地/开源偏好：BGE、M3E 一类模型

这些只是示例。用户如果要“最新推荐”，先查官方资料。

### 第 3 轮：选向量数据库

优先根据规模、过滤需求、部署方式来定。

默认启发：
- 学习与本地验证：Chroma
- 中小规模生产：Qdrant
- 大规模与企业化：Milvus 或托管服务

### 第 4 轮：决定是否加 Rerank

当检索结果不稳定、文档量较大、或需要更精确排序时，再引入 Rerank。不要默认一上来就加复杂度。

### 第 5 轮：生成层与 Prompt

定义回答格式、引用方式、是否要求来源可追溯，再讨论模型。

### 第 6 轮：测试与评估

至少覆盖三类验证：

- 功能验证：能不能检索到相关片段并生成可用回答
- 质量验证：相关性、完整性、引用准确性
- 性能验证：响应时间、吞吐、成本

### 第 7 轮：调优

先从检索侧调，再到生成侧调，最后做系统层优化。顺序通常是：

1. 分块与 Top-K
2. Embedding 与 Rerank
3. Prompt 与上下文拼接
4. 缓存、批处理、并发

## 什么时候给参考实现

只有在以下情况才给代码或参考实现：

- 用户明确表示要开始写代码
- 当前决策已经确定
- 参考实现能帮助用户完成当前这一轮的小任务

不要在用户还没选型前就输出整套代码骨架。

## 实战结束时的成果

一个完整实战通常应产出：

1. 一份明确的技术选型结论
2. 一个能跑通的最小 RAG 流程
3. 一组基础测试样例
4. 下一步优化清单
