# 实战中心说明

这个文件补充说明平台中的 `实战中心`，不再把“实战模式”当作独立产品壳。

## 实战中心目标

实战中心面向“我要做一个 RAG 系统”的请求。核心原则是逐步推进：每一轮只处理一个关键决策，避免一次把完整方案全部抛给用户。

场景列表统一见 [catalog.md](catalog.md)。

## 进入实战前先确认

正式给方案前，先尽量补齐这些约束：

- 场景：文档问答、客服机器人、企业搜索、垂直领域等
- 数据：PDF、Markdown、网页、数据库，或混合数据
- 语言：中文、英文、多语言
- 规模：几百条、几万条、还是百万级
- 预算：优先效果、优先成本，还是可本地部署
- 部署方式：本地验证、云服务、企业内网
- 成功标准：更准、更快、更便宜，还是更可解释

如果用户没给全，不要一次问很多。优先问当前决策最关键的 1 到 2 个约束。

## 每轮输出协议

每一轮都按下面格式组织：

1. `当前决策`
2. `推荐方案`
3. `取舍理由`
4. `最小实现任务`
5. `进入下一轮的条件`

## 推荐推进顺序

1. 场景与成功标准
2. Embedding 与分块策略
3. 向量数据库
4. 是否加 Rerank
5. 生成层与引用策略
6. 测试与评估
7. 调优

## 什么时候切到 RAG Lab

以下情况优先建议进入实验，而不是继续空谈：

- 两种 embedding 纠结不下
- 不确定是否需要 rerank
- chunk size / top-k 影响不清楚
- 召回质量与延迟存在明显冲突

## 实战结束时的成果

一个完整实战通常应产出：

1. 一份明确的技术选型结论
2. 一个能跑通的最小 RAG 流程
3. 一组基础测试样例
4. 下一步优化清单
