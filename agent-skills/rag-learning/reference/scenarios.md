# RAG 实战场景详细说明

本文件只保留场景的详细说明、典型用例和起步建议。

场景编号、名称、难度、预计时间、推荐起点统一维护在 [catalog.md](catalog.md)。如果目录信息有变更，先改 `catalog.md`，再检查这里的详细说明是否仍然成立。

## 使用方式

- 需要快速选场景时，优先看 [catalog.md](catalog.md)
- 已经选定场景，想看进一步解释时，再看本文件
- 这里只给默认起步建议，不把它当成固定标准答案
- 涉及模型、数据库、供应商能力等内容时，这些都只是候选示例；如果用户要“最新推荐”，应先核实官方资料

## 01 文档问答系统

适合上传 PDF、Word、Markdown 等文档后，围绕文档内容做精确问答。

**典型用例**
- 上传产品手册后，询问“如何安装 XX”
- 上传法律合同后，询问“保密条款是什么”
- 上传技术文档后，询问“某个参数如何配置”

**起步建议**
| 组件 | 候选示例 | 说明 |
|------|----------|------|
| Embedding | BGE-large-zh / text-embedding-3-small | 中文支持或成本平衡 |
| 向量数据库 | Chroma / Qdrant | 本地验证或小规模生产 |
| 分块策略 | 语义分块 | 尽量保留句子和段落完整性 |
| Rerank | bge-reranker 一类模型 | 提升最终排序质量 |
| 生成模型 | GPT-4o-mini 一类模型 | 效果与成本平衡 |

**常见难点**
- 文档结构复杂，导致分块质量不稳定
- 长文档中细粒度信息召回不足
- 引用来源不准确，影响可追溯性

## 02 客服机器人

适合基于产品知识库、帮助中心或企业内部知识做多轮问答。

**典型用例**
- 电商客服：订单、物流、退换货问题
- SaaS 客服：功能使用、配置问题
- 企业内部客服：HR、行政、IT 问答

**起步建议**
| 组件 | 候选示例 | 说明 |
|------|----------|------|
| Embedding | text-embedding-3-small / BGE-large-zh | 响应速度或中文适配 |
| 向量数据库 | Pinecone / Milvus / Qdrant | 托管、规模化或通用生产 |
| 分块策略 | 固定大小分块 | 统一、高效，便于快速起步 |
| Rerank | Cohere Rerank / 开源中文重排 | 对复杂查询有帮助 |
| 生成模型 | GPT-4o 一类对话模型 | 提升对话自然度 |

**常见难点**
- 多轮对话上下文和检索上下文互相干扰
- 用户问题口语化，召回结果不稳定
- 回答需要同时兼顾语气、准确性和业务规则

## 03 企业知识库搜索

适合企业内部海量文档、制度、规范、流程说明等搜索场景。

**典型用例**
- 搜索“年假政策”
- 搜索“报销流程”
- 搜索“会议室预定方法”

**起步建议**
| 组件 | 候选示例 | 说明 |
|------|----------|------|
| Embedding | BGE-large-zh | 中文语义理解与成本控制 |
| 向量数据库 | Milvus / Qdrant | 适合较大规模和过滤需求 |
| 分块策略 | 文档结构分块 | 尽量保留标题和层级 |
| Rerank | 中文重排模型 | 提升精确匹配效果 |
| 生成模型 | GPT-4o-mini / Claude 3.5 一类模型 | 在可读性和成本间平衡 |

**常见难点**
- 权限隔离、过滤条件和元数据设计
- 标题层级丢失导致引用结果不自然
- 海量数据下的检索延迟和成本

## 04 垂直领域问答

适合医疗、金融、法律等高专业度、高准确性要求的场景。

**典型用例**
- 医疗：症状咨询、用药说明
- 金融：理财产品、风险解释
- 法律：法条检索、合同条款理解

**起步建议**
| 组件 | 候选示例 | 说明 |
|------|----------|------|
| Embedding | BGE-large-zh / 领域适配模型 | 强调领域匹配度 |
| 向量数据库 | Milvus / Qdrant | 支持规模和过滤 |
| 分块策略 | 语义分块 | 尽量保留专业上下文 |
| Rerank | 领域重排模型 | 对术语与精确排序更友好 |
| 生成模型 | Claude 3.5 / GPT-4o 一类长文本模型 | 适合复杂上下文理解 |

**常见难点**
- 专业术语召回不准
- 错误回答的业务风险高
- 需要更严格的来源引用与人工复核流程

## 05 多模态 RAG

适合图文混合检索，例如产品目录、教程文档、博物馆导览等。

**典型用例**
- 搜索“红色连衣裙”，返回商品图片和描述
- 搜索“如何组装”，返回步骤图片和文字说明
- 搜索文物名称，返回图片和背景介绍

**起步建议**
| 组件 | 候选示例 | 说明 |
|------|----------|------|
| Embedding | CLIP / 多模态嵌入模型 | 将图文映射到可检索空间 |
| 向量数据库 | Milvus / Pinecone | 适合多模态向量存储 |
| 分块策略 | 多模态分块 | 保持图文关联 |
| Rerank | 跨模态 Rerank | 帮助图文相关性排序 |
| 生成模型 | GPT-4o / Claude 3.5 一类多模态模型 | 便于整合图文信息 |

**常见难点**
- 图和文的切分边界不好定义
- 图文对齐不稳，导致召回看似相关但实际答非所问
- 评估标准比纯文本更复杂

## 06 实时问答

适合低延迟和流式输出要求较强的场景。

**典型用例**
- 直播问答
- 会议助手
- 在线客服边输入边显示答案

**起步建议**
| 组件 | 候选示例 | 说明 |
|------|----------|------|
| Embedding | text-embedding-3-small 一类快速模型 | 优先响应速度 |
| 向量数据库 | Qdrant | 低延迟起步常见选择 |
| 分块策略 | 小块分块 | 缩短检索时间 |
| Rerank | 可选 | 先看延迟预算再决定 |
| 生成模型 | 支持 streaming 的模型 | 便于流式输出 |
| 额外组件 | WebSocket / SSE | 实时推送结果 |

**常见难点**
- 检索、生成、前端流式展示的链路时延叠加
- 引入 Rerank 后可能明显变慢
- 用户对“速度快”和“答案准”的预期经常冲突

## 07 混合检索 RAG

适合同时需要语义理解和关键词精确匹配的场景。

**典型用例**
- 搜索引擎
- 代码搜索
- 产品搜索与规格匹配

**起步建议**
| 组件 | 候选示例 | 说明 |
|------|----------|------|
| Embedding | BGE-large-zh | 提供语义召回能力 |
| 关键词检索 | BM25 / Elasticsearch | 提供精确匹配能力 |
| 向量数据库 | Qdrant / Weaviate | 适合混合搜索能力扩展 |
| 分块策略 | 语义分块 | 保留文本完整性 |
| Rerank | bge-reranker 一类模型 | 融合多路结果 |
| 生成模型 | GPT-4o-mini 一类模型 | 控制整体成本 |

**常见难点**
- 两种召回结果如何融合
- 精确匹配和语义相似的权重难调
- 评估时需要同时看召回率和排序质量

## 场景选择建议

- 初学者优先从 `01 文档问答系统` 开始
- 需要多轮交互时优先看 `02 客服机器人`
- 文档量大且需要过滤时优先看 `03 企业知识库搜索`
- 专业场景优先看 `04 垂直领域问答`
- 想做高级演示或专项能力时看 `05 / 06 / 07`
