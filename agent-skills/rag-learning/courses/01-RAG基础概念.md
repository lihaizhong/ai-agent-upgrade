## 1. 什么是 RAG？

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将**检索系统**与**生成模型**结合的技术架构。

**核心思想**：让 LLM 在生成回答之前，先从外部知识库中检索相关信息，利用这些信息来增强回答的质量和准确性。

### 为什么需要 RAG？

| 问题 | RAG 如何解决 |
|------|-------------|
| LLM 知识过时 | 实时检索最新知识 |
| 幻觉（Hallucination） | 基于真实检索内容生成 |
| 领域知识缺乏 | 接入专业领域知识库 |
| 回答不可追溯 | 提供信息来源 |
| 训练成本高 | 无需重新训练 |

## 2. RAG 的核心流程

```
┌─────────────────────────────────────────────────────────────┐
│                      RAG 流程图                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐ │
│   │  文档   │───▶│  索引   │───▶│  检索   │───▶│  生成   │ │
│   │  输入   │    │  构建   │    │  查询   │    │  回答   │ │
│   └─────────┘    └─────────┘    └─────────┘    └─────────┘ │
│        │              │              │              │       │
│        ▼              ▼              ▼              ▼       │
│   PDF/DOC/HTML   分块+向量化    语义相似度    LLM 生成     │
│                  存入向量库      Top-K 匹配                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 索引阶段（Indexing）

1. **文档加载**：读取 PDF、Word、Markdown、TXT 等格式文档
2. **文本分块**：将长文档切分为较小的文本块（Chunk）
3. **向量化**：使用 Embedding 模型将文本块转换为向量
4. **存储**：将向量和原始文本存入向量数据库

### 检索阶段（Retrieval）

1. **查询向量化**：将用户问题转换为向量
2. **相似度计算**：在向量数据库中查找最相似的文本块
3. **结果排序**：返回 Top-K 个最相关的结果

### 生成阶段（Generation）

1. **上下文组装**：将检索结果与原始问题组合
2. **Prompt 构建**：创建包含检索内容的提示词
3. **LLM 生成**：调用大语言模型生成回答
4. **后处理**：整理输出，返回最终答案

## 3. RAG vs 其他方法

### RAG vs 纯 LLM

| 对比项 | 纯 LLM | RAG |
|--------|--------|-----|
| 知识来源 | 训练数据 | 可更新的知识库 |
| 回答准确性 | 可能 hallucination | 基于检索内容 |
| 知识时效性 | 取决于训练时间 | 实时更新 |
| 可追溯性 | 不可追溯 | 可追溯到来源 |
| 成本 | API 调用成本 | 额外检索成本 |

### RAG vs Fine-tuning（微调）

| 对比项 | RAG | Fine-tuning |
|--------|-----|-------------|
| 知识更新 | 实时（更新知识库） | 需重新训练 |
| 成本 | 较低 | 较高（GPU 训练） |
| 适用场景 | 知识问答、实时性 | 风格迁移、任务优化 |
| 可解释性 | 高（检索来源） | 低（隐含参数） |
| 延迟 | 较高（检索+生成） | 较低（直接生成） |

### 何时选择 RAG？

**适合 RAG**：
- 需要最新知识（新闻、实时数据）
- 需要可追溯的回答
- 知识库需要频繁更新
- 成本敏感，无法承担训练成本

**适合微调**：
- 需要特定语言风格或格式
- 任务高度特定化
- 有充足的训练预算
- 数据相对稳定

**两者结合**：
- RAG 提供知识检索
- 微调优化生成风格
- 效果最佳

### 组件详解

| 组件 | 作用 | 技术选型 |
|------|------|----------|
| 文档加载器 | 读取各种格式文档 | LangChain、Unstructured |
| 分块器 | 将文档切分为小块 | 固定大小、语义分块 |
| Embedding 模型 | 将文本转为向量 | OpenAI、BGE、M3E |
| 向量数据库 | 存储和检索向量 | Milvus、Pinecone、Chroma |
| 检索器 | 查找相关文档 | 向量检索、关键词检索 |
| Rerank 模型 | 优化结果排序 | Cohere、BAAI |
| LLM | 生成最终回答 | GPT-4、Claude |

## 5. 简单 RAG 代码示例

```python
from langchain.document_loaders import TextLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import Chroma
from langchain.chat_models import ChatOpenAI
from langchain.chains import RetrievalQA

# 1. 加载文档
loader = TextLoader("knowledge.txt")
documents = loader.load()

# 2. 文档分块
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50
)
chunks = splitter.split_documents(documents)

# 3. 向量化存储
embeddings = OpenAIEmbeddings()
vectorstore = Chroma.from_documents(chunks, embeddings)

# 4. 创建检索链
llm = ChatOpenAI(model="gpt-4o")
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    retriever=vectorstore.as_retriever()
)

# 5. 问答
question = "什么是 RAG？"
answer = qa_chain.run(question)
print(answer)
```
