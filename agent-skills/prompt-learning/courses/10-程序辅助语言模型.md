# 程序辅助语言模型 (PAL)

## 核心概念

PAL 让 AI **生成程序代码**来解决需要精确计算的问题，然后将代码交给运行时执行。就像让AI写出计算步骤，再让计算器执行。

> **一句话理解**：AI写代码，程序来执行，精确无误。

## 工作原理

```
问题（如数学计算）
  ↓
AI生成程序代码
  ↓
代码执行器运行代码
  ↓
精确结果
```

## 实用示例

### 日期计算

**问题**：
```
今天是2023年2月27日，我25年前出生，
出生日期是？（MM/DD/YYYY格式）
```

**AI生成代码**：
```python
from datetime import datetime
from dateutil.relativedelta import relativedelta

today = datetime(2023, 2, 27)
born = today - relativedelta(years=25)
print(born.strftime('%m/%d/%Y'))
```

**执行结果**：02/27/1998

## 核心优势

1. **精确计算**：避免AI计算错误
2. **可验证**：代码可审查
3. **处理复杂逻辑**：利用编程能力
4. **数据结构支持**：数组、字典等

## 适用场景

✅ 数学计算
✅ 日期时间处理
✅ 数据分析
✅ 算法实现

## 与思维链的对比

| PAL | 思维链 |
|-----|--------|
| 代码执行 | 文本推理 |
| 精确无误 | 可能有计算错误 |
| 可验证 | 难以验证 |

## 何时使用 / 何时不该使用

### ✅ 推荐使用
- **问题可形式化**：能明确写成程序步骤或公式
- **需要高精度计算**：文本推理容易算错
- **适合借助编程库**：如日期、统计、解析、图算法

### ❌ 不推荐使用
- **问题本质是开放式判断**：代码执行无法替代主观分析
- **没有安全执行环境**：生成代码但无法安全运行
- **任务主要靠外部知识**：PAL 擅长算，不擅长补事实

## 常见误区

- 以为“代码能跑”就等于“结论一定对”，忽略了程序本身也可能写错。
- 对不适合编程表达的问题也强行生成代码。
- 没有隔离执行环境，直接运行不可信代码。
