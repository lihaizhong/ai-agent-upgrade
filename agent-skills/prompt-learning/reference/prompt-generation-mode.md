# Prompt Lab 参考

这份文档只保留 Prompt Lab 的澄清和优化策略。固定流程、必填槽位、审查清单和校验规则都以 `scripts lab ...` 的输出为准。

## 目标

把模糊需求收敛成结构化 prompt，并借机教会学习者为什么这些槽位会影响结果。

## 推荐流程

1. 读取 `--workflow` 和 `--interview-blueprint`。
2. 通过追问补齐 `task`、`input`、`output_format`、`constraints`、`quality_bar`。
3. 调用 `--validate-slots` 校验槽位完整性，不完整就继续补。
4. 槽位完整后再生成初稿。
5. 读取审查清单，只修改未通过项。
6. 调用 `--validate-draft` 校验草稿和审查记录。
7. 给学生解释每次修改背后的原因，不要只扔最终 prompt。

## 澄清时怎么问

优先围绕五个槽位追问：

- `task`: AI 最终要完成什么任务？
- `input`: 它会拿到哪些输入材料？
- `output_format`: 输出要长什么样？
- `constraints`: 有哪些限制、禁区或硬规则？
- `quality_bar`: 什么结果才算合格？

追问原则：

- 一次补最缺的 1 到 2 个槽位，不要一次抛出长问卷。
- 问题要贴近用户原话，不要机械复读字段名。
- 能从上下文推出的内容就直接归纳给用户确认，不必全都追问。
- 这是教学环节，不是纯代写环节；澄清时要顺带解释“为什么要问这个”。
- Prompt Lab 默认不是选择器驱动流程；除非是少量明确分支动作，否则不要把槽位澄清强行改成结构化菜单。

## 生成时的取舍

- 简单任务优先用简单结构，不要默认堆技术名词。
- 只有需要示例、推理或组合技术时，才加对应结构。
- 约束要可执行，输出格式要可判定。

## 审查重点

审查时优先看：

- 任务目标是否单一明确
- 输入定义是否足够
- 输出格式是否可直接验收
- 约束是否可执行
- 是否说明了失败或边界处理
- 是否存在重复或冲突指令

## 常见失误

- 没补齐槽位就直接生成。
- 跳过 `--validate-slots` 或 `--validate-draft`，让流程重新退回纯人工约束。
- 审查时整段重写，而不是只改失败项。
- 需求很简单，却写出一大段复杂角色设定和冗余约束。
- 输出格式写得很模糊，导致无法稳定复用。
- 把用户当成协作者一起“脑暴”，却没有完成教学解释。
